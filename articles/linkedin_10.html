I'm six times slower. With the same model. On the same CPU.

So I do what any engineer would do. I profile.

And here's where the story gets interesting.

Everyone assumes matrix multiplication is the bottleneck. It's a neural network. Millions of multiplications. That's where the time goes. Right?

Wrong.

Matrix multiplication was 6% of total time.

The real killers were the operations nobody writes blog posts about. Layer normalization. GELU activation. Memory layout conversions.

The boring stuff. The glue code between the important parts.

Except in this case — the glue was the important part.

My engine gave the right answer at the wrong speed.

Then I made it fast — and it started giving wrong answers.

Cosine similarity with the reference model: 0.067. Should be 1.0. That's not a rounding error. That's garbage.

I spent three weeks comparing 286 tensors, layer by layer, between my C code and a NumPy reference. Every number. Every dimension. Every sign bit.

Found seven bugs. Each one invisible on its own. Together — catastrophic.

The sneakiest: the ONNX graph showed a Conv1x1 taking the input tensor. But the input to that conv was actually a constant. Pre-computed. Never changes. I was recomputing it every frame, introducing drift that accumulated through seventeen layers.

Seven fixes. 0.067 became 1.000.

Speed is glamorous. Accuracy is where you earn your sleep.

There's a processor instruction called vpdpbusd. It multiplies thirty-two 8-bit integers and accumulates the result in one clock cycle.

Most developers have never heard of it. Most never will.

I spent two weeks learning how to use it. Then I spent a month fighting a saturation bug in its cousin instruction that corrupts results when values exceed 32,767.

This is what low-level optimization looks like. Not clever algorithms. Not elegant abstractions. Reading processor manuals. Counting clock cycles. Finding that your multiply-add silently overflows on the 4,097th element.

Glamorous? No. But it's how you beat a 28-megabyte framework with 148 kilobytes of C.

The last two milliseconds almost broke me.

I had gone from 24ms to 5ms in about six weeks. Felt like a god. Every day brought a new breakthrough. New SIMD kernel here, remove a memory copy there, batch some operations — boom, another millisecond gone.

Then I hit 5ms and the wall appeared.

I tried everything. Assembly language. JIT compilation. Different compilers. Link-time optimization. Profile-guided optimization. Thread pool tuning. Operation fusion. Cache line alignment.

Nothing worked. Four months of nothing.

Then I found it. The processor on my machine had AVX-512. A wider SIMD instruction set. Nobody told me. I had to read the CPU flags.

Rewrote the kernels. 5ms became 3ms.

The lesson: when you hit a wall, sometimes the answer isn't to push harder. It's to look at the wall from a different angle.
